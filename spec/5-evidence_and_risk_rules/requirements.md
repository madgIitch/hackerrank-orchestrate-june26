# 5 · Suficiencia de evidencia y risk flags — Requisitos

- name: `evidence_and_risk_rules` · priority: P1 · sdd: true
- aprobado por: peorr · 2026-06-19T17:58:03.567Z

## Contexto

Implementar reglas deterministas para evidence_standard_met, valid_image y risk_flags visuales y de historial.

## Requisitos funcionales

R1. Existe un módulo code/evidence_rules.py con funciones puras y deterministas que operan sobre datos estructurados ya existentes: fila normalizada del modelo, EvidenceLookupResult, UserHistory, image_ids válidos, errores de carga de imagen y, opcionalmente, triage visual_summary solo como texto explicativo. No se añade ningún campo nuevo al contrato de 10 campos del modelo ni al output.csv.
R2. compute_evidence_standard_met no hace matching por keywords ni comparación semántica de texto libre. Si EvidenceLookupResult.requirements está vacío, devuelve evidence_standard_met=false y un reason que empieza exactamente por `No matching evidence requirement found for claim_object=<claim_object> issue_type=<issue_type>.`. Si hay requirements, conserva el booleano normalizado del modelo salvo que reglas estructurales lo invaliden: sin imagen usable, supporting_image_ids inválidos para supported/contradicted, imágenes faltantes que impiden verificar, o valid_image=false.
R3. compute_history_risk_flags propaga history_flags válidos del CSV y, si history_flags='none', deriva flags con estos umbrales conservadores: user_history_risk si rejected_claim >= 2, o last_90_days_claim_count >= 4, o manual_review_claim >= 2 y past_claim_count >= 5; manual_review_required si rejected_claim >= 3, o last_90_days_claim_count >= 5, o manual_review_claim >= 3. Valores fuera del enum se descartan. Los flags de historial nunca cambian claim_status ni evidence_standard_met.
R4. valid_image=true si al menos una imagen cargada, nítida, relevante y usable muestra el objeto o la pieza reclamada con suficiente contexto para revisar la condición. valid_image=false solo cuando ninguna imagen cargada permite evaluar el objeto/parte reclamada, o cuando todas las imágenes relevantes están inutilizables, son del objeto equivocado, no muestran la parte/daño necesario o tienen wrong_angle que impide inspección.
R5. En multi-imagen, risk_flags es la unión deduplicada y ordenada de flags canónicos por fila, serializada con punto y coma en la columna existente. No hay prefijos por image_id ni formato nuevo. El detalle por imagen puede explicarse en evidence_standard_met_reason o claim_status_justification.
R6. Las reglas deterministas se aplican como post-proceso en pipeline.py tras merge_output_row, filter_supporting_image_ids y validate_output_row, y antes de la precedencia final de evidence_standard_met=false. El orden canónico final es: merge/validate, filter_supporting_image_ids, apply_evidence_rules, apply_evidence_precedence, apply_history_risk_flags. Si apply_evidence_rules produce evidence_standard_met=false, apply_evidence_precedence degrada claim_status a not_enough_information y supporting_image_ids=none. Ningún paso posterior puede elevar claim_status usando historial.
R7. Los tests unitarios en tests/test_evidence_rules.py son deterministas sin modelo y cubren: requirements vacíos con prefijo exacto de reason; requirements presentes + modelo true preserva true; requirements presentes + imagen no usable fuerza false; history_flags='none' bajo umbral sin flags; history_flags='none' sobre umbrales deriva user_history_risk/manual_review_required; history_flags explícitos se propagan; flag fuera de enum se descarta; multi-imagen con una imagen usable mantiene valid_image=true y fusiona risk_flags.
R8. Los tests existentes de features 3/4 que cubren fallback, validación de enums, supporting_image_ids, evidence_standard_met, risk_flags, valid_image y escritura a tmp_path se actualizan solo para el comportamiento documentado, sin reducir cobertura.
R9. evaluation_report.md se actualiza con distribución de evidence_standard_met true/false sobre sample_claims.csv, frecuencia de cada risk_flag individual, ratio de valid_image=false y notas de errores o limitaciones observadas. Cuando exista ground truth comparable en sample_claims.csv, el reporte compara contra esos valores; cuando no exista, lo marca como diagnóstico descriptivo.

## Restricciones

- **error_states:** Fuente primaria: history_flags del CSV, parseados por punto y coma, mapeados a enums risk_flags; valores inválidos descartados. Si history_flags='none', derivar flags con umbrales conservadores: user_history_risk si rejected_claim >= 2, o last_90_days_claim_count >= 4, o (manual_review_claim >= 2 y past_claim_count >= 5); manual_review_required si rejected_claim >= 3, o last_90_days_claim_count >= 5, o manual_review_claim >= 3. Estos flags nunca elevan claim_status ni evidence_standard_met; solo agregan riesgo y justificación. Umbrales y lógica de derivación se documentan en DECISIONS.md.
- **auth_secrets:** La feature opera sobre CSV locales ya cargados por feature 2 y sobre output del modelo existente. No introduce nuevas dependencias externas ni credenciales.
- **rollback_compat:** Se añade code/evidence_rules.py con funciones puras; pipeline.py lo llama como post-proceso tras la normalización/validación existente, sin alterar el orden de pasos previos. parser_validator.py solo se extiende si hace falta conectar la normalización con las reglas, no como lugar principal de la lógica. Los tests existentes de features 3/4 se actualizan para reflejar el nuevo paso post-proceso, conservando cobertura de fallback, enums, degradación de claim_status, supporting_image_ids válidos y escritura a tmp_path.

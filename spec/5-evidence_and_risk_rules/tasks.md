# 5 - Suficiencia de evidencia y risk flags - Tareas

Checklist de implementacion. El agente marca [x] al completar; los gates verifican.

- [x] (T1) Existe un modulo code/evidence_rules.py con funciones puras y deterministas que operan sobre datos estructurados ya existentes: fila normalizada del modelo, EvidenceLookupResult, UserHistory, image_ids validos, errores de carga de imagen y, opcionalmente, triage visual_summary solo como texto explicativo. No se anade ningun campo nuevo al contrato de 10 campos del modelo ni al output.csv. R1
- [x] (T2) compute_evidence_standard_met no hace matching por keywords ni comparacion semantica de texto libre. Si EvidenceLookupResult.requirements esta vacio, devuelve evidence_standard_met=false y un reason que empieza exactamente por `No matching evidence requirement found for claim_object=<claim_object> issue_type=<issue_type>.`. Si hay requirements, conserva el booleano normalizado del modelo salvo que reglas estructurales lo invaliden: sin imagen usable, supporting_image_ids invalidos para supported/contradicted, imagenes faltantes que impiden verificar, o valid_image=false. R2
- [x] (T3) compute_history_risk_flags propaga history_flags validos del CSV y, si history_flags='none', deriva flags con estos umbrales conservadores: user_history_risk si rejected_claim >= 2, o last_90_days_claim_count >= 4, o manual_review_claim >= 2 y past_claim_count >= 5; manual_review_required si rejected_claim >= 3, o last_90_days_claim_count >= 5, o manual_review_claim >= 3. Valores fuera del enum se descartan. Los flags de historial nunca cambian claim_status ni evidence_standard_met. R3
- [x] (T4) valid_image=true si al menos una imagen cargada, nitida, relevante y usable muestra el objeto o la pieza reclamada con suficiente contexto para revisar la condicion. valid_image=false solo cuando ninguna imagen cargada permite evaluar el objeto/parte reclamada, o cuando todas las imagenes relevantes estan inutilizables, son del objeto equivocado, no muestran la parte/dano necesario o tienen wrong_angle que impide inspeccion. R4
- [x] (T5) En multi-imagen, risk_flags es la union deduplicada y ordenada de flags canonicos por fila, serializada con punto y coma en la columna existente. No hay prefijos por image_id ni formato nuevo. El detalle por imagen puede explicarse en evidence_standard_met_reason o claim_status_justification. R5
- [x] (T6) Las reglas deterministas se aplican como post-proceso en pipeline.py tras merge_output_row, filter_supporting_image_ids y validate_output_row, y antes de la precedencia final de evidence_standard_met=false. El orden canonico final es: merge/validate, filter_supporting_image_ids, apply_evidence_rules, apply_evidence_precedence, apply_history_risk_flags. Si apply_evidence_rules produce evidence_standard_met=false, apply_evidence_precedence degrada claim_status a not_enough_information y supporting_image_ids=none. Ningun paso posterior puede elevar claim_status usando historial. R6
- [x] (T7) Los tests unitarios en tests/test_evidence_rules.py son deterministas sin modelo y cubren: requirements vacios con prefijo exacto de reason; requirements presentes + modelo true preserva true; requirements presentes + imagen no usable fuerza false; history_flags='none' bajo umbral sin flags; history_flags='none' sobre umbrales deriva user_history_risk/manual_review_required; history_flags explicitos se propagan; flag fuera de enum se descarta; multi-imagen con una imagen usable mantiene valid_image=true y fusiona risk_flags. R7
- [x] (T8) Los tests existentes de features 3/4 que cubren fallback, validacion de enums, supporting_image_ids, evidence_standard_met, risk_flags, valid_image y escritura a tmp_path se actualizan solo para el comportamiento documentado, sin reducir cobertura. R8
- [x] (T9) evaluation_report.md se actualiza con distribucion de evidence_standard_met true/false sobre sample_claims.csv, frecuencia de cada risk_flag individual, ratio de valid_image=false y notas de errores o limitaciones observadas. Cuando exista ground truth comparable en sample_claims.csv, el reporte compara contra esos valores; cuando no exista, lo marca como diagnostico descriptivo. R9
